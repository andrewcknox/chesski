## Terminology

The line-selection doc calls them "type-1" and "type-2" frontiers. In conversation with the user, use these names instead:

- **PTM-F** = Player-To-Move Frontier (type-1). Your turn, walking through stored prep, no stored move at this position.
- **OTM-F** = Opponent-To-Move Frontier (type-2). Opponent on move, popular Lichess reply exists, no stored response to that reply.

Both are gaps in the prep that training tries to close.

## The three algorithms — DO NOT conflate

Line generation runs three independent algorithms that live in `src/lib/autosuggest.ts`. The most common failure mode is "fixing" one while thinking you're editing another.

1. **Frontier selection** (`findTopFrontier` / `findTopUnansweredOpponentMove`) — picks the *next gap to study*. Ranked **purely by `games`** (Lichess game count at the frontier's childFen). Do not re-introduce `weight` into selection logic — that bug has been fixed twice already.
2. **User-move selection** (`pickYourMove` and friends) — picks what *you* play at each ply of the generated line. Sources: player-book → masters → lichess-2000 → engine, in user-configured order; player-book picks pass through the same per-move WDL gate as the quality gate (shared `judgeCandidateAtFen` helper). **`games` plays no role here.**
3. **Opponent-move-in-line** (`pickOpponentMoves`) — picks what the *opponent* plays after the frontier. Top Lichess Explorer popularity. Also scaffolds every above-threshold reply into the repertoire so future training has slots.

## Hot invariants — burn-it-down list

Verbatim from [docs/line-selection.md](docs/line-selection.md) — these are recurring failure modes:

- **Do not** sort frontiers by `weight × games` or any combined score.
- **Do not** re-introduce `weight` as a tiebreaker in any of the three frontier sort sites.
- **Do not** add a synthetic depth cap on total line length.
- **Do not** use `games` in `pickYourMove` or any user-move selection.
- **Do not** let line extension follow a source game past `learnLineDepth` user moves.
- **Do not** add recursive DFS in `src/lib/viewLines.ts` — cyclic repertoire data infinite-loops.
- **Do not** flip eval sign in `LineEvalPanel` — it is white-positive by design.
- **Do not** delete the `Copy failure report` / `Copy line report` buttons.
- **Do not** route line-gen engine evals through `fetchCloudEval`. The Lichess cloud endpoint ignores `depth`. All depth-sensitive evals use `fetchLocalEval`.
- **Do not** negative-cache local Stockfish failures (`localEvalCache.set(cacheKey, null)`). Only cache successes; failures get retried.
- **Do not** reintroduce the blocked-frontier-on-gate-failure mechanism. Single attempt per `generateLearnLine` call; if a line fails, surface it. `markFrontierBlocked` is for stale-root only.
- **Do not** let `putFrontiers` overwrite a non-`'open'` existing status when the search rediscovers a frontier. Removing the `keepExisting` guard re-introduces the duplicate-lines bug (answered frontiers silently re-opened on every rebuild).
- **Do not** gate purely on cp-loss. The quality gate is per-user-move WDL (`maxWdlExpectedScoreDrop` / `maxWdlLossDelta`); cp-loss is a fallback for the WDL-absent case only.
- **Do not** publish short lines except for `game-over` termination (mate, stalemate, threefold, fifty-move, insufficient material). Other early exits throw `FrontierGenerationError`.

## Scope is the most fragile area

The frontier-index scope filter (`frontierInScope`) requires two checks: scope-key match *and* path-through-startFen. The second check is what prevents "ancestor frontiers" (positions upstream of the scope endpoint that got tagged with the scope's key during discovery) from being offered to line generation. See **"Scope" → "What 'in scope' means for a frontier"** in [docs/line-selection.md](docs/line-selection.md) for the why and the worked example.

If you're about to weaken `frontierInScope`, make `pickYourMoveDirect`'s engine fallback scope-aware, or move discovery so it starts at `scope.startFen` — **stop and re-read the scope section.** Those three "fixes" have each been tried; only the path-through-startFen filter is correct. (Historical note: the engine fallback used to be a rollout-style multi-PV simulator called `pickEngineRolloutMove`, removed in 2026-05 after being fooled by its own optimism; the current single-call depth-22 picker is `pickYourMoveDirect`'s `getEngine` closure.)
